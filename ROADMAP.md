# 项目路线图

Cursor Skin Manager 当前专注于 Windows 本地光标皮肤管理。路线图用于说明方向和边界，不承诺固定发布日期；具体实现仍需要 Issue、设计确认、测试和代码审查。

## 当前状态

- 当前稳定版本：[`v0.1.11`](https://github.com/Myming15/cursor-skin-manager/releases/tag/v0.1.11)
- 支持平台：Windows 10/11，主要验证目标为 x64
- 下一个产生应用行为或二进制变化的版本：`0.1.12`
- 版本记录：[`CHANGELOG.md`](CHANGELOG.md)
- 开发边界：[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md)

当前 MVP 已形成完整闭环：导入本地皮肤、识别 CUR/ANI/INF、预览 15 个角色、应用到当前用户、替换和重新分配文件、恢复默认以及安全删除本地副本。

## 状态定义

| 状态 | 含义 |
| --- | --- |
| 计划 | 已确认属于项目方向，会按社区执行步骤或独立 Issue 推进，但不代表固定发布日期 |
| 研究中 | 有明确价值，但方案、风险、优先级或兼容性尚未确认 |
| 不在范围内 | 当前 MVP 明确不实现，不应直接提交代码或导航占位 |

## 计划

### 社区与质量基础

这些工作已经在 [`docs/社区开源建设执行步骤.md`](docs/社区开源建设执行步骤.md) 中确认：

- 建立 Windows GitHub Actions，执行 npm、TypeScript、Vitest、Rust 格式、测试和编译检查。
- 增加 Dependabot、依赖审计和 CodeQL 等基础安全检查。
- 保护 `main` 和公开版本标签，阻止未经检查的强制推送与标签移动。
- 建立可重复的 Windows 安装版、便携版、SHA-256 和 Draft Release 流程。
- 整理发布、回滚、Issue 分类和维护节奏。

纯文档、模板和仓库设置不提升应用版本。自动化流程一旦生成与当前公开版本不同的应用二进制，就必须使用新的版本号。

### 下一次应用版本

下一次功能、Bug 修复、依赖更新或构建变化如果需要发布二进制，版本从 `0.1.12` 开始。具体内容在对应 Issue 被接受前不预先承诺；优先级遵循：

1. 防止用户原始文件、内部皮肤和 Windows 光标配置损坏。
2. 修复跨电脑启动、CUR/ANI/INF 兼容性和中文路径问题。
3. 改善导入、预览、应用、替换、分配和删除主流程的性能与可访问性。
4. 在不扩大 MVP 的前提下减少维护成本。

## 研究中

以下项目是候选优化，不等于已经排期：

| 方向 | 当前问题 | 进入实现前需要确认 |
| --- | --- | --- |
| 导入失败清理 | 某些早期复制或解析失败可能留下未被库引用的内部目录 | 失败注入、目录归属校验和恢复测试 |
| 删除与清空事务 | 文件删除早于最终 JSON 写入，后续持久化失败无法恢复已删除文件 | 可恢复暂存目录、回滚策略和磁盘空间影响 |
| 注册表部分写入恢复 | 15 个角色逐项写入，不是跨全部值的原子事务 | 写入前快照、失败恢复和 Windows 刷新行为 |
| 数据格式版本 | `library.json` 当前没有显式 `schemaVersion` | 版本字段、逐级迁移、备份和降级规则 |
| 大型皮肤性能 | 大量 CUR/ANI 的扫描、解码和 data URL 传输可能增加启动时间 | 基准样本、惰性加载和缓存失效策略 |
| ANI 动画预览 | 当前界面只展示首个可解析帧 | 帧时序解析、资源占用和无障碍降级 |
| Windows ARM64 | 主要逻辑预期可移植，但尚未完成同等范围验证 | ARM64 工具链、安装包和真实设备测试 |
| Windows 代码签名 | 当前公开包未商业签名 | 证书类型、年度成本、密钥保护和时间戳服务 |

架构已知约束的完整说明见 [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md#当前约束)。高风险改动应拆成独立 PR，并提供失败回滚和旧数据兼容测试。

## 不在范围内

当前不计划实现：

- 在线皮肤市场、远程皮肤下载或内容推荐。
- 账号、登录、收藏、标签、评分、社交和云同步。
- macOS、Linux 或跨平台光标应用。
- 修改所有 Windows 用户、系统级强制安装或要求管理员权限的方案。
- Windows 主题、壁纸、图标包、声音和其他桌面美化管理。
- 未经用户明确操作自动替换系统光标。
- 在仓库中收集或分发来源不明、无授权的第三方光标素材。

范围外建议仍可以通过 Feature Request 说明问题和使用场景，但在维护者明确调整路线图前，不应直接提交实现。

## 版本与支持策略

- 项目处于 `0.x` 阶段，保持主次版本不变时使用补丁号连续递增。
- 当前只为最新公开 Release 提供安全修复，旧版本用户应先升级复现。
- 文档、社区模板和 GitHub 设置变化不单独提升应用版本。
- 任何改变应用行为、依赖、构建结果或生成新二进制的修改都必须提升版本。
- 发布时同步更新四个版本源、`CHANGELOG.md`、Release Notes 和校验文件。
- 已公开标签不可移动或覆盖；发布错误通过新版本修复。

四个版本源是 `package.json`、`package-lock.json`、`src-tauri/Cargo.toml` 和 `src-tauri/tauri.conf.json`。详细规则见 [`docs/版本管理.md`](docs/版本管理.md)，安全支持范围见 [`SECURITY.md`](SECURITY.md)。

## 如何参与

1. 先搜索现有 Issue 和路线图，确认问题没有重复。
2. Bug 和兼容性问题可以直接按 Issue Form 提交最小复现。
3. 较大功能或高风险数据变更先创建 Feature Request，等待范围确认。
4. 实现前阅读 [`CONTRIBUTING.md`](CONTRIBUTING.md)、[`docs/DEVELOPMENT.md`](docs/DEVELOPMENT.md) 和 [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md)。

路线图会根据已验证问题、维护能力和社区反馈调整。调整状态时，应在相关 Issue 或 Pull Request 中保留原因。

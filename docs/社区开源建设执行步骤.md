# Cursor Skin Manager 社区开源建设 AI 执行步骤

## 1. 文档目的

本文档用于指导 AI 将 Cursor Skin Manager 从“公开源码并提供下载”逐步建设为“适合外部用户反馈、贡献者参与和维护者长期运营”的社区开源项目。

本文档只定义执行步骤，不代表这些步骤已经完成。后续执行时应一次只推进一个步骤，完成验证并向用户汇报后，再进入下一步。

当前基线：

- 项目版本：`0.1.11`
- 技术栈：Tauri 2、React、TypeScript、Rust
- 默认分支：`main`
- 开源许可：MIT License
- 已有内容：用户 README、Windows 安装版、便携版、SHA-256、GitHub Release、前端测试和 Rust 测试
- 当前维护者：Myming15

## 2. 本轮范围

本计划包括：

1. 社区贡献文档。
2. Issue、PR 和支持模板。
3. 安全策略与漏洞报告机制。
4. 开发者文档。
5. CI、质量门禁和依赖安全。
6. 分支与标签保护。
7. 版本、路线图和自动化发布。
8. Windows 代码签名方案。
9. 社区问题分流和长期维护机制。

本计划明确不包括：

- 第三方素材、角色、截图和光标包的版权处理。
- 在线市场、账号、收藏、标签和云同步等非 MVP 功能。
- 与社区建设无关的产品功能扩展。
- 未经用户确认的商业证书购买或付费服务开通。

## 3. AI 执行规则

1. 开始每一步前，先读取当前工作区和远程仓库状态，不假设文档仍与代码完全一致。
2. 不覆盖用户未提交的修改，不回滚与当前步骤无关的文件。
3. 每一步只修改该步骤直接涉及的文件和 GitHub 设置。
4. 每一步完成后都要执行对应验证，并报告新增文件、设置变化、测试结果和剩余风险。
5. 社区文档、模板和仓库设置本身不提升应用版本；只要生成了新的应用二进制或修改了应用行为，就必须从 `0.1.12` 开始递增版本。
6. GitHub Actions 权限使用最小权限，任何令牌、证书、密码和签名材料都不得写入仓库或日志。
7. 首次上线的自动化发布应生成 Draft Release，由维护者人工验证后再发布，不能直接自动推送正式安装包。
8. 在 CI 检查名称稳定之前，不设置依赖这些名称的强制分支规则。
9. 不修改已经公开发布的版本标签。需要修复发布内容时创建新版本，不移动旧标签。
10. 每个阶段优先通过独立分支和 Pull Request 完成，避免一次提交混合所有社区建设内容。

## 7. 阶段三：建立自动质量门禁

### 步骤 8：建立基础 CI

**做什么**

创建 Windows GitHub Actions 工作流，在 Push 和 Pull Request 时执行：

- 固定 Node LTS 版本并执行 `npm ci`。
- 前端组件测试。
- TypeScript 和 Vite 生产构建。
- Rust 格式检查。
- Rust 测试。
- 基础 Clippy 检查，初期可以作为非阻塞检查。

CI 应采用最小权限、依赖缓存和可读的 Job 名称。普通 PR 不构建约 200 MB 的离线安装包，避免浪费时间和 Actions 存储。

**解决什么问题**

解决“只在维护者电脑能运行”、错误代码直接进入主分支，以及贡献者无法知道失败原因的问题。

**主要产物**

- `.github/workflows/ci.yml`
- README 中的 CI 状态入口，可在工作流稳定后再增加徽章。

**验收标准**

- 干净分支上的前端和 Rust 检查全部通过。
- 人为制造一个测试或格式错误时，CI 能正确失败。
- CI 不依赖本机缓存、私有目录或交互式窗口。

### 步骤 9：清理并启用严格代码质量检查

**做什么**

在基础 CI 稳定后建立统一质量标准：

- 清理现有 Clippy 警告。
- 将 `cargo clippy --all-targets -- -D warnings` 设为阻塞检查。
- 为 React 和 TypeScript 增加 ESLint 规则。
- 选择并固定前端格式化规则。
- 增加统一的 `lint`、`format:check` 等项目脚本。
- 对共享行为和高风险 Windows 操作逐步补充测试。

**解决什么问题**

解决代码风格漂移、警告长期累积、不同贡献者使用不同检查方式，以及 CI 只能发现编译错误的问题。

**主要产物**

- 干净的 Clippy 结果。
- 前端 Lint 和格式配置。
- CI 中稳定的质量检查名称。

**验收标准**

- 本地和 CI 使用相同命令得到相同结果。
- 新增警告、格式问题或明显的 TypeScript 规则问题会阻止合并。
- 清理警告过程中不改变未授权的产品行为。

### 步骤 10：建立依赖与供应链安全

**做什么**

创建 `.github/dependabot.yml`，覆盖：

- npm。
- Cargo。
- GitHub Actions。

同时配置或评估：

- Dependabot Alerts。
- Dependabot Security Updates。
- npm 依赖审计。
- Rust 漏洞和许可证审计。
- TypeScript 代码扫描。
- GitHub Actions 依赖固定与最小权限。
- 第三方依赖许可证清单。

Dependabot 更新建议按周运行，并限制同时打开的 PR 数量，避免维护者被更新请求淹没。

**解决什么问题**

解决依赖长期不更新、已知漏洞无人发现、Action 依赖失控和第三方许可证不兼容的问题。

**主要产物**

- `.github/dependabot.yml`
- 已启用的 Dependabot 安全功能。
- 依赖与许可证审计结果。

**验收标准**

- GitHub 能识别 npm、Cargo 和 Actions 三类依赖。
- Dependabot 检查能够正常运行且不会一次创建过多 PR。
- 高严重性依赖漏洞有明确处理流程。

## 8. 阶段四：保护主分支和发布历史

### 步骤 11：配置分支与标签 Ruleset

**做什么**

在 CI 检查名称稳定后，为 `main` 配置规则：

- 外部修改必须通过 Pull Request。
- 合并前必须通过前端和 Rust CI。
- 阻止 Force Push 和分支删除。
- 要求解决所有审查对话。
- 单维护者阶段可保留受控的管理员紧急绕过权限。

为 `v*` 发布标签配置规则：

- 阻止修改或强制移动已经发布的版本标签。
- 限制可以创建发布标签的角色。

**解决什么问题**

解决未经检查的代码直接进入主分支、历史被重写、已发布标签被移动，以及 Release 无法追溯的问题。

**主要产物**

- `main` 分支 Ruleset。
- `v*` 标签 Ruleset。

**验收标准**

- 普通分支不能绕过 CI 直接合并。
- Force Push 到 `main` 和已发布标签会被拒绝。
- 维护者仍有记录清晰的紧急处理方式。

## 9. 阶段五：规范 Windows 发布

### 步骤 12：建立自动化 Release

**做什么**

创建独立发布工作流，建议通过版本标签或手动触发：

- 验证 `package.json`、`package-lock.json`、`Cargo.toml`、`tauri.conf.json` 和版本文档一致。
- 执行全部前端和 Rust 检查。
- 在 Windows Runner 构建 Tauri Release。
- 生成安装版、便携版和 SHA-256。
- 统一无空格、稳定的小写连字符文件名。
- 创建 Draft Release 并上传资产。
- 根据 `CHANGELOG.md` 生成 Release Notes。
- 维护者下载并冒烟验证后，人工发布为 Latest Release。

首次实现时先通过手动 Dry Run 或预发布标签验证，不直接覆盖现有 `v0.1.11`。

**解决什么问题**

解决手工打包步骤遗漏、版本号不一致、Release 文件名被 GitHub改写、校验文件失配和错误二进制直接公开的问题。

**主要产物**

- `.github/workflows/release.yml`
- 自动生成的 Draft Release。
- 可重复的发布日志和构建产物。

**验收标准**

- 同一提交可以重复得到结构一致的发布文件。
- 安装版、便携版和校验清单名称完全一致。
- SHA-256 可以校验线上下载文件。
- 没有签名密钥或 GitHub Token 出现在日志和 Artifact 中。

### 步骤 13：评估并接入 Windows 代码签名

**做什么**

按以下顺序评估：

1. SignPath Foundation 等符合条件的开源项目签名方案。
2. Microsoft Store 分发。
3. Azure Artifact Signing。
4. 传统 OV 代码签名证书。

确认方案后：

- 通过 OIDC、受保护环境或 GitHub Secrets 接入签名。
- 不把 `.pfx`、密码或私钥提交到仓库。
- 在生成 SHA-256 前完成 EXE 和安装包签名。
- 使用系统工具验证签名、时间戳和发布者名称。
- 在 README 和 Release Notes 中说明签名状态。

**解决什么问题**

解决 Windows 显示未知发布者、企业环境阻止未签名程序、用户难以判断安装包来源的问题。

**主要产物**

- 已确认的签名方案。
- 受保护的签名流程。
- 已签名的发布文件和验证记录。

**验收标准**

- Windows 文件属性能显示预期发布者。
- 签名和时间戳验证通过。
- 新版本更换文件哈希后仍可能出现 SmartScreen 信誉提示，文档不得承诺签名后一定没有提示。

### 步骤 14：记录发布和回滚流程

**做什么**

创建 `docs/RELEASING.md`，记录：

- 版本选择和升级规则。
- 发布前检查清单。
- 创建标签和 Draft Release 的顺序。
- 安装版与便携版的冒烟测试。
- 签名和 SHA-256 验证。
- Release 发布和公告步骤。
- 发现严重问题后的撤回、标记和修复版本流程。
- 禁止移动已经公开的版本标签。

**解决什么问题**

解决发布知识只存在维护者记忆中、紧急修复时误删历史，以及后续维护者无法复现发布的问题。

**主要产物**

- `docs/RELEASING.md`
- 可复用的发布检查清单。

**验收标准**

- 另一名维护者仅按照文档即可完成一次 Draft Release。
- 回滚流程不会覆盖旧标签或静默替换已下载文件。

## 10. 阶段六：建立社区运营节奏

### 步骤 15：建立标签和问题分流规则

**做什么**

建立最小标签集合：

- `bug`
- `enhancement`
- `documentation`
- `security`
- `cursor-format`
- `windows`
- `good first issue`
- `help wanted`
- `needs reproduction`
- `needs decision`
- `wontfix`

定义维护者处理流程：确认、分类、补充信息、接受、关闭和发布关联。选择少量边界清楚的任务标记为 `good first issue`。

**解决什么问题**

解决 Issue 数量增加后无法检索、重复问题长期堆积、新贡献者找不到合适任务的问题。

**主要产物**

- GitHub 标签集合。
- Issue 分流说明。
- 首批适合贡献者的任务。

**验收标准**

- 新 Issue 可以在一次分流中获得类型和状态标签。
- `good first issue` 不涉及注册表、文件删除或发布密钥等高风险区域。

### 步骤 16：执行社区发布前总验收

**做什么**

完成一次从访客、贡献者、维护者和最终用户四个视角的验收：

- 访客能理解项目、许可、下载方式和反馈入口。
- 贡献者能从干净克隆完成安装、测试和 PR。
- 维护者能处理 Issue、私密漏洞和依赖更新。
- 最终用户能下载、校验、安装、运行和找到日志。
- CI、Ruleset、Dependabot 和 Draft Release 流程均实际运行一次。
- 检查仓库中没有密钥、个人路径、构建目录和超大二进制。

**解决什么问题**

解决文件看似齐全但链接失效、流程从未演练、权限设置互相阻塞，以及真正有贡献者时才发现无法使用的问题。

**主要产物**

- 一份社区发布验收记录。
- 尚未解决问题及风险清单。
- 对外发布公告草案。

**验收标准**

- GitHub Community Profile 的核心项目完整。
- 一次测试 PR 能从创建到 CI 通过并合并。
- 一次测试漏洞报告能私密到达维护者。
- 一次发布 Dry Run 能生成正确资产但不会自动公开。

### 步骤 17：建立长期维护周期

**做什么**

建立轻量维护节奏：

| 周期 | 维护任务 |
| --- | --- |
| 每周 | 分流 Issue、检查安全报告、处理 Dependabot PR |
| 每月 | 检查依赖、CI 稳定性、失效链接和待确认路线图 |
| 每次发布 | 更新版本、CHANGELOG、测试、签名、校验和 Release Notes |
| 每季度 | 复查支持版本、路线图、社区规则和维护者权限 |

**解决什么问题**

解决社区文件创建后长期失效、依赖告警堆积、路线图失真和维护工作完全依赖临时记忆的问题。

**主要产物**

- 可执行的维护周期。
- 定期复查记录或 GitHub Project 任务。

**验收标准**

- 每类维护任务都有负责人或默认维护者。
- 不设置维护者无法长期兑现的硬性响应 SLA。

## 11. 推荐执行批次

为降低一次性改动风险，建议按以下批次执行：

| 批次 | 包含步骤 | 建议提交或 PR |
| --- | --- | --- |
| A：社区基础 | 0 至 5 | `docs: add community contribution and support files` |
| B：开发文档 | 6 至 7 | `docs: add development, architecture, and roadmap guides` |
| C：基础自动化 | 8 | `ci: add Windows validation workflow` |
| D：质量与安全 | 9 至 10 | 分成质量清理和依赖安全两个 PR |
| E：仓库保护 | 11 | GitHub 设置变更，单独记录前后状态 |
| F：发布体系 | 12 至 14 | 自动发布、签名和发布文档分别审查 |
| G：社区上线 | 15 至 17 | 标签、验收和维护节奏 |

## 12. 总体验收标准

全部步骤完成后，项目应满足：

1. 用户、贡献者、安全研究人员和维护者都有明确入口。
2. 外部贡献可以在干净环境构建和测试。
3. 所有 PR 在进入 `main` 前通过自动检查。
4. 依赖漏洞和版本更新可以被持续发现。
5. `main` 和版本标签受到保护。
6. Release 可以稳定生成安装版、便携版和匹配的 SHA-256。
7. 签名材料、令牌和用户隐私不会进入仓库或日志。
8. 版本、CHANGELOG、README 和 Release Notes 保持一致。
9. 社区运营规则足够清楚，同时不过度承诺维护者资源。
10. 项目仍保持本地 Windows 光标管理 MVP 定位，不因社区建设扩展非目标功能。

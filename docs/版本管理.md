# 版本管理

本文定义 Cursor Skin Manager 的版本来源、递增条件和公开标签规则。面向用户的变化见 [`CHANGELOG.md`](../CHANGELOG.md)，未来方向见 [`ROADMAP.md`](../ROADMAP.md)。

## 当前版本

- 当前稳定版本：`0.1.11`
- 首次公开标签：`v0.1.11`
- 公开日期：2026-07-13
- 下一次功能、Bug 修复、依赖或构建变化产生新二进制时：从 `0.1.12` 开始
- 当前未发布变化只有社区与开发文档，应用版本继续保持 `0.1.11`

项目仍处于 `0.x` 开发阶段。补丁版本必须连续递增，不能重复使用已经公开的版本号。

## 版本源

正式构建前必须确认以下四处完全一致：

| 文件 | 字段 |
| --- | --- |
| `package.json` | 根对象 `version` |
| `package-lock.json` | 根包与 `packages[""]` 的 `version` |
| `src-tauri/Cargo.toml` | `[package] version` |
| `src-tauri/tauri.conf.json` | 根对象 `version` |

设置页通过 Tauri 运行时 API 读取版本，不在 React 界面中硬编码正式版本号。`Cargo.lock` 中本项目包版本由 Cargo 更新，也需要在发布检查中确认。

## 何时提升版本

以下变化必须提升版本：

- 新增、删除或改变用户可见功能。
- 修复会改变应用运行行为的 Bug。
- 修改 Rust、React、Tauri 权限、配置或 Windows 集成行为。
- 更新依赖、构建参数、图标或资源，并生成与当前 Release 不同的二进制。
- 重新打包安装版或便携版并准备公开分发，即使改动看起来很小。

以下变化可以不提升应用版本：

- README、贡献指南、路线图和其他纯文档修改。
- Issue Form、Pull Request 模板、标签和仓库设置。
- 不生成应用二进制、也不改变构建结果的 CI 文档或检查配置。

判断原则是：只要生成并准备分发新的应用二进制，就使用新版本号。不能用同一个公开版本号覆盖旧安装包、便携版或校验文件。

## 发布同步规则

一次正式发布需要同步：

1. 更新四个版本源并检查 `Cargo.lock`。
2. 将 `CHANGELOG.md` 的“未发布”内容移动到带日期的新版本章节。
3. 执行前端测试、生产构建、Rust 格式、测试和编译检查。
4. 构建安装版和便携版，生成匹配的 SHA-256 校验文件。
5. 从将要发布的提交创建 `v<version>` 标签。
6. 创建 Draft Release，上传资产并完成 Windows 冒烟验证。
7. 人工发布为 Latest Release，并确认 README 下载入口正确。

公开标签不可强制移动、删除后重建或指向新的提交。发现发布错误时，应撤下有问题的资产或标记说明，并通过新的补丁版本修复。

## 历史口径

- `v0.1.11` 是仓库中首个具有可验证公开标签和 Release 的稳定版本。
- `0.1.10` 是角色替换与未分配文件重新分配的开发里程碑，没有独立公开标签，其功能包含在 `v0.1.11` 中。
- `0.1.1` 是应用图标清晰度改进的开发里程碑，没有独立公开标签，其变化包含在 `v0.1.11` 中。
- 没有标签或 Release 的开发版本不列为可下载历史版本，也不补造发布日期。

详细变化统一记录在 [`CHANGELOG.md`](../CHANGELOG.md)，不要在多个文档中分别维护互相冲突的版本清单。

## 支持策略

项目只为最新公开 Release 接受安全修复，不为旧版本长期回移补丁。发布新版本后，应同步更新 [`SECURITY.md`](../SECURITY.md) 的支持表和 README 当前稳定版本。

文档、版本源、Git 标签、GitHub Release 和下载资产必须指向同一个版本。任何不一致都应阻止发布。
